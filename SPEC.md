# YouTube Metrics CLI — Спецификация проекта


## Архитектура и поток данных

### 1. Паттерны
- **Strategy**: каждый отчёт наследуется от абстрактного `Report` и реализует `filter()`, `sort()`, `get_columns()`.
- **Registry**: `ReportRegistry` хранит маппинг `имя_отчёта -> класс`. Регистрация через декоратор `@ReportRegistry.register("name")`. Позволяет добавлять новые отчёты без правки ядра.

### 2. Поток выполнения
1. `main.py` парсит аргументы через `cli/parser.py`
2. `data/loader.py` читает все переданные CSV через `csv.DictReader`, валидирует колонки и типы, возвращает `List[VideoMetric]`
3. `reports/registry.py` по имени `--report` создаёт экземпляр нужного отчёта
4. Отчёт применяет `filter()` → `sort()`
5. `utils/output.py` рендерит результат через `tabulate` в `stdout`

### 3. Ключевые компоненты
- `VideoMetric`: `@dataclass(frozen=True)` с валидацией диапазонов в `__post_init__`
- `load_videos_from_files()`: агрегирует данные из нескольких файлов, бросает `FileNotFoundError`/`ValueError` при проблемах
- `ReportRegistry`: потокобезопасный dict-кеш + декоратор регистрации
- `main()`: только оркестрация + `try/except` с возвратом кодов `0`/`1`/`2`

## Стратегия тестирования
| Модуль | Что покрывать |
|--------|--------------|
| `cli/parser.py` | Обязательные аргументы, отсутствие, неизвестный `--report` |
| `data/loader.py` | Валидный CSV, отсутствие колонок, нечисловые значения, несуществующий путь |
| `reports/clickbait.py` | Фильтрация по границам, сортировка по убыванию CTR, пустой вход |
| `reports/registry.py` | Регистрация декоратором, `get("unknown")` → ошибка, `list_available()` |
| `main.py` | Интеграционный тест: корректный запуск, код выхода `0`, ошибки в `stderr` |
